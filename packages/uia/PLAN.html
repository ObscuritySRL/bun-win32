BUN_UIA — Implementation Plan: @bun-win32/uia + bun-uia

For agentic workers: this plan is designed for end-to-end execution by Claude (Opus 4.8, ultracode) via a /goal. Track phases with TaskCreate/TaskUpdate. Execute tasks in order; each task is bite-sized, verified, and committed before moving on. Never leave the repo in a broken state. The goal is complete when bun-uia@1.0.0 and @bun-win32/uia@1.0.0 are live on npm, installable, and the post-publish acceptance test (Phase 15, S15.7) passes — a clean-room bun add bun-uia that finds Notepad by title, types into it, reads the text back through the UIA tree, and asserts.

This document follows the house standard for product plans set by packages/gpu/PLAN.html (BUN_GPU). It reproduces that structure: ground-truth dossier → research-while-building doctrine → execute-time competitive intelligence with a parity matrix ("everything the incumbents do, only better" — gated at release) → bite-sized verified phases → real tests → AI-first docs → one-OTP release → clean-room acceptance. BUN_GPU Appendix F is the cloning guide; this is the second plan in the series.

Goal: Productize the proven UI-Automation COM-vtable demo (packages/all/example/uia-automation.ts — it launches Calculator, walks its live UIA tree, and physically clicks buttons to compute 5+3=8) into a first-class, AI-first package @bun-win32/uia: a Playwright-for-Windows-desktop — query the accessibility tree by name/control-type/automationId, invoke controls, type, read values, wait-for-element, and serialize the tree to JSON for LLM agents — then publish it plus an unscoped bun-uia alias to npm, with real tests at every step.

Architecture: A non-binding "product" package (precedent: @bun-win32/terminal) of flat root modules built over existing binding packages — @bun-win32/{core, combase, ole32, oleaut32, uiautomationcore, user32, oleacc} — with one small new binding-work item: the in-process COM activation trio (CoCreateInstance, CoInitializeEx, CLSIDFromString, CoUninitialize) is currently bound in NO package (the demo inline-dlopens combase.dll), so Phase 2 adds it to a binding package. The spine is the IUIAutomation COM client (the proven query+invoke path, matching every incumbent's mental model) driven through a cast-free vtable invoker (vcall); the already-shipped flat uiautomationcore C-API (102 proven exports) and oleacc MSAA are first-class secondary engines for capabilities where they are faster or VARIANT-free. No marshaling layer, no native build step, no npm dependencies outside the workspace.

Tech stack: Bun (bun:ffi), TypeScript strict, Biome formatter, COM vtable FFI into UIAutomationCore.dll via combase.dll, user32 SendInput, oleacc MSAA fallback. Windows-only, Bun-only — stated proudly, not apologetically.

⚡ The performance doctrine (UIA is cross-process — read before every hot-path phase)

A desktop-automation library that is slower than spawning a process has no reason to exist. But UIA's performance model is the opposite of a GPU engine's tight inner loop: the dominant cost is the cross-process COM round-trip — every get_Current* property read and every FindAll marshals across the boundary into the target app's UIA provider. The incumbents pay this through an HTTP WebDriver hop (WinAppDriver listens on 127.0.0.1:4723) or a separate runtime; bun-uia is in-process COM — already orders of magnitude lower latency — but it can still be made slow by N naive round-trips. Treat every number below as a floor, not a finish line — while building, keep hunting for wins this plan never anticipated (see the generative pass, G.3). The rules (the FFI/JS levers were measured on the BUN_GPU authoring host 2026-06-09 — Bun runs on JavaScriptCore, not V8; re-verify per host, treat V8 folklore as suspect until measured; the UIA-side numbers come from this plan's own benchmarks, never copied; full data in Appendix G):

  • Cache or be slow — the #1 UIA lever. The proven demo reads each element's control-type and name with two separate cross-process vcalls per element (get_CurrentControlType, get_CurrentName) — N elements ⇒ 2N round-trips. The documented fix is IUIAutomationCacheRequest + the *BuildCache method family: one round-trip prefetches many properties and patterns for a whole subtree. Phase 7 builds this; the high-level find/tree paths MUST use it. MSDN: IUIAutomationTreeWalker navigation "is not as efficient as locating an element using FindAll or FindFirst" — and even FindAll without a cache pays per-property round-trips afterward.
  • Never FindAll(TreeScope_Descendants) from the desktop root. The desktop has thousands of nodes; MSDN warns this risks a stack overflow and is pathologically slow. Always scope to a window element (ElementFromHandle) and prefer TreeScope_Children + targeted descent, or a cached FindAll bounded to the subtree you need.
  • Mind the vcall hot path. The invoker builds a template-string key (`${method}|${argTypes.join(',')}`) and a Map.get on EVERY COM call. For tree serialization over thousands of nodes this is in the inner loop. Microbench it (S7.x); if it shows, key by the resolved method pointer — but only with a measurement, and NEVER change vcall's argument contract (argTypes EXCLUDE the implicit this; the invoker prepends FFIType.u64 — a spurious leading u64 segfaults, ground truth).
  • Reuse, never re-allocate, in any per-element / per-poll loop. A Buffer.alloc per iteration measured ~1064 ns (~1600× vs reuse); a fresh DataView per parse is +20 ns. The out-pointer buffers that receive each get_Current* result, the waitFor poll's scratch, and the tree-walk's element-array reads must use hoisted, reused buffers — but honor the GC-window rule (assemble structs immediately before the blocking call; never cache a small Buffer's .ptr across allocations — it relocates).
  • BSTR readback is a per-character loop today. getBstr in the demo copies SysStringLen × 2 bytes one at a time via read.u8. For a tree of thousands of named nodes this dominates. Bulk-copy the UTF-16 region in one operation (and remember SysStringLen returns CHARACTERS, not bytes; copy BEFORE SysFreeString — reading after free is use-after-free).
  • Pace polling correctly. waitFor retries on an interval. Async Bun.sleep(1) quantizes to ~15.6 ms; Bun.sleepSync(1) does not (~1.5 ms) — but a sync sleep blocks Bun's only thread. Choose the model deliberately (the proven demo uses Bun.sleepSync(400) for its window-find retry); never busy-spin a tight while loop burning a core.
  • Avoid bigint in the hot path; minimize FFI crossings. The FFI floor is ~5 ns/call; a u64→bigint return adds ~10 ns; a fresh BigInt() arg per call is ~16 ns (hoist constant handle args — the IUIAutomation pointer, the cached condition — out of loops). Pass Number(x) as Pointer (the sanctioned narrowing).
  • Every hot path gets an isolated microbenchmark (Bun.nanoseconds(), 200k+ warm-up, dead-code sink, median ns/op — the terminal.bench.ts convention): a single vcall property read, a cached-vs-uncached subtree walk (the headline number), a waitFor poll iteration, BSTR decode per node, INPUT packing. Benchmarks DRIVE optimization, not just the README table. A perf baseline is recorded in Phase 0 and gated at release (S14.6) exactly like a tsc error.
  • Prove it, don't hope — Bun ships bun:jsc. Because this is Bun-only, every hot path can be verified optimized: numberOfDFGCompiles(fn) > 0 (timing optimized code, not the interpreter), profile(cb) tier breakdown (assert FTL/DFG dominates the tree-walk decode), reoptimizationRetryCount(fn) (climbing = type-unstable vcall sites), jscDescribe(obj) (Structure X-ray of the Element result objects), and heapStats().objectTypeCounts deltas (gate per-element allocation in a tree walk). Full toolkit in Appendix G.4/G.5 — use it in every microbench and audit.
  • Don't satisfice — beating the baseline is the job. Correctness plus the known levers is the START of a hot-path audit, never the end. Once a path is correct, profile() it to find the ACTUAL dominant cost (often the cross-process round-trip count, the BSTR loop, or the vcall key build), form a hypothesis, microbench an alternative, keep it only if it wins AND stays correct, cast-free, and readable — then go again. Can the algorithm change (one cached subtree fetch instead of N property reads; the flat uiautomationcore path instead of COM where it is VARIANT-free and faster), not just the constant factor? The S14.6 gate enforces "no regression"; the standing goal is to beat the prior baseline every phase. Any lever you discover that this plan didn't list — prove it with the G.4 tools and add it to Appendix G.5. Never stop at "fast enough."

S6.1Create packages/uia/input.ts — the x64 INPUT packer NEW. Hand-pack the 40-byte INPUT (DWORD type @0, 4 bytes padding, union @8: MOUSEINPUT 32B / KEYBDINPUT 24B), and call User32.SendInput(count, buffer.ptr, 40). cbSize MUST be 40 on x64 or SendInput silently injects nothing (the #1 trap). Reference layout (verify offsets by running): 
function packKeyboardInput(out: Buffer, offset: number, vk: number, scan: number, flags: number): void {
  out.writeUInt32LE(INPUT_KEYBOARD, offset + 0);   // type
  // offset+4: padding
  out.writeUInt16LE(vk, offset + 8);               // KEYBDINPUT.wVk
  out.writeUInt16LE(scan, offset + 10);            // wScan
  out.writeUInt32LE(flags, offset + 12);           // dwFlags
  out.writeUInt32LE(0, offset + 16);               // time
  out.writeBigUInt64LE(0n, offset + 24);           // dwExtraInfo (ULONG_PTR) — 24, not 20 (8-byte align)
}

Phase 8 — The high-level facade: find/findAll/waitFor — the Playwright API (flagship)

Purpose: the ergonomics that make this "Playwright for the desktop." This is where competitive parity is won (Appendix D): selector-based query, auto-retry/wait, and a fluent action chain — typed end to end, zero casts. No Windows-desktop npm library offers waitFor.

API design commitments (frozen here, validated against the Phase 0 competitive research):
  • const app = uia.attach(title | hWnd | { process }) → a Window (an Element scoped to a top-level window). uia.focused() / uia.fromPoint(x,y) are the other entries.
  • app.find(selector) / app.findAll(selector) — cached, scoped to the window (never the desktop root). Returns Element / Element[].
  • await app.waitFor(selector, { timeout = 5000, interval }) — poll find until it matches or times out; the killer feature for flaky native UIs and agents. Paced per the perf doctrine (no busy-spin; deliberate sleep model). Times out with an error that quotes the selector + the window + the nearest candidates.
  • Fluent actions on the returned Element: .invoke(), .click(), .setValue(text), .type(text), .toggle(), .select(), .expand(), .scrollIntoView(), .focus(), .text(), .screenshot(path) — each from Phases 5/6/9.

  • Lifecycle is explicit but forgiving: a using app = uia.attach(...) (or app.dispose()) releases element pointers; a process-exit handler calls uninitialize(). Document the apartment-affinity caveat (tolerate UIA_E_ELEMENTNOTAVAILABLE).
S8.1Build the facade in automation.ts/element.ts NEW. attach, Window, waitFor, the fluent chain. waitFor re-runs find (cached) on an interval; on each miss it releases the partial results (no leak across polls). D.3 — The "why now": the computer-use wave wants exactly this

The frontier desktop agents ground actions in screenshots and the literature calls it fragile and expensive; the fix they are converging on is the accessibility tree (UIA), but existing a11y pipelines are slow and token-heavy — precisely the seam a fast, typed, in-process UIA query+invoke API fills.

  • Screenshot grounding is the acknowledged bottleneck. Anthropic computer use is "still experimental — at times cumbersome and error-prone"; its reference impl downscales to XGA and maps coordinates back ("training Claude to count pixels accurately was critical"). OSWorld screenshot-only ≈ 14.9% vs 72.4% human. (anthropic.com/news/3-5-models-and-computer-use; anthropic-quickstarts/computer-use-demo)
  • VLMs can't reliably locate clickable elements from a bitmap — OmniParser's stated motivation. (microsoft.github.io/OmniParser)
  • Microsoft UFO2 fuses the UIA tree FIRST, vision second, explicitly to fix "fragile screenshot-based interaction." (arXiv 2504.14603)
  • But a11y pipelines are slow + bloated: OSWorld-Human reports a11y-tree builds taking 3–26 s and adding "thousands more tokens per step." (arXiv 2506.16042) → a fast in-process build-time is a real, measurable win (S0.8b / benchmark).
  • No live Node/Bun UIA client exists: NodeRT is dead + wrong-namespace (15/wk), win32-api has no UIA, ffi-napi is abandoned ("PLEASE ARCHIVE THIS REPO"), koffi leaves you hand-writing COM vtables, edge-js drags in .NET, Playwright _electron can't reach native windows. bun-uia is the only zero-install, typed, in-process Bun/npm path to the live UIA tree.

Parity feature this creates: the agent tool-call adapter + tree→JSON (10b #1, Phase 9) — "drop-in computer-use grounding for Windows desktop." Honest limit: UIA doesn't cover every app (owner-draw/games), so this is a complement to screenshots, not a replacement — and the "fast" claim must be a measured number, not an assertion (Appendix D open question, gated by the benchmark).

D.4 — Capabilities NONE of them offer (the moat — each needs a proof artifact)

  • Zero-install, typed, in-process a11y-tree query+invoke on Bun/npm — every other UIA path is a native addon (NodeRT/@bright-fish), a 2-daemon server (WinAppDriver+Appium), or a foreign runtime (FlaUI/.NET, pywinauto/Python).
  • Playwright-style waitFor auto-retry + web-test-runner-class assertions for the desktop (10b #2).
  • a11y-tree → JSON agent grounding with a measured sub-second build time, beating the OSWorld 3–26 s reference (Phase 9 / agent-grounding.ts).
  • Three engines in one facade: IUIAutomation COM client + the flat uiautomationcore VARIANT-free fast path + oleacc MSAA fallback — breadth only this repo can offer (it already ships all three).
  • Crash-safe polling input observation (no foreign-thread hook) — the iohook/uiohook hazard sidestepped (10b #6).
  • AI-first documentation contract (AI.md) — no incumbent has anything comparable.

G.3 — The discipline

  1. Baseline (S0.8b) records the property-read + naive-subtree-walk floors on the release host before any work.
  2. Every hot-path phase (3, 7, 8) ends with a microbenchmark recording ns/op (or ms/subtree) and asserting zero per-element allocation.
  3. Release gate (S14.6) re-runs everything; a >~10% regression vs baseline OR any forbidden hot-loop pattern blocks release.
  4. README numbers (ms/µs/tokens) come only from the gated run — never copied from this appendix.
  5. Generative pass (the real spirit — do this every hot-path phase). An audit does not just apply G.5's levers and check the gate. profile() the path to locate the true hot spot, then actively try at least one optimization this plan did NOT anticipate — a deeper cache prefetch, the flat-API path instead of COM for a read, a coarser tree filter, a reused decode view — measure it, fold any proven win back into G.5. For UIA specifically: the biggest wins are algorithmic (fewer round-trips: one cached BuildCache instead of N get_Current*; scope tighter; prune the agent profile harder) — chase those before constant-factor JS tuning. "Meets baseline" is the floor; beating it is the job.
  6. Methodology (verified on Bun 1.4.0/JSC): Bun.nanoseconds(); warm ≥200k iterations; assert numberOfDFGCompiles(fn) > 0 and abort if 0 (else you're timing LLInt/Baseline); folded sink; vary inputs to defeat hoisting; gcAndSweep() between trials; median of ≥5. For the cross-process walk, wall-clock IS the honest metric (it includes the round-trip the JS-side tools can't see) — pair it with a heapStats alloc-delta to prove the JS side is clean.
