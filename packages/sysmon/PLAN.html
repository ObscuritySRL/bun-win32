⚡ The performance doctrine (this product IS performance — read before every hot-path phase)

bun-sysmon's entire reason to exist is "the same data in microseconds, not 100 ms." A correct-but-slow metric defeats the product. Performance is therefore a standing, always-on audit, not a final benchmark — every hot-path phase ends by analyzing its inner loop down to allocations, FFI-boundary crossings, and CPU cycles. Treat every number below as a floor, not a finish line — while building, keep actively hunting for wins this plan never anticipated; the listed levers are where to start, not where to stop (see the generative pass, G.3). The rules (measured on the authoring machine 2026-06-09 — Bun runs on JavaScriptCore, not V8; re-verify per host, and treat any V8 folklore as suspect until measured; full data in Appendix G):

  • ZERO allocation inside any sampling / decode / per-row loop. The single biggest lever: Buffer.alloc(19 KB) per tick measured ~1064 ns; reusing one buffer is ~free (~1600×). Allocate every scratch buffer, DataView, and typed-array view ONCE at sampler construction and reuse it every tick (the proven core-scope.ts pattern: perfBuf/procBuf allocated outside the loop, grow-only realloc). A fresh DataView per parse costs +20 ns; hold {buffer, dataView, u8} as persistent sampler fields. No per-row object/array/closure/template-string in the hot path either — budget the GC cost of returning ProcessInfo[] (object-per-row + Date) and offer a struct-of-arrays path for high-frequency consumers.
  • Avoid bigint in the hot path. A u64 FILETIME delta as Number(a - b) measured ~41 ns; reading the two u32 halves and doing number math is ~3 ns (13×). Per-core CPU times swept as two-u32-halves are 8× faster than DataView.getBigInt64. Reserve bigint for handles/pointers (correct + unavoidable) and magnitudes that truly exceed 2^53. Caveat: the absolute FILETIME exceeds 2^53 — only the small delta fits the Number path; subtract the hi/lo halves before widening.
  • Cache buffer.ptr once per loop. The .ptr getter (core extensions) re-runs ptr(this) on every access (~7.3 ns); a cached const is ~free — ~19% off a real QueryPerformanceCounter(buf.ptr) loop. Never write nativeFn(buf.ptr) inside a per-tick loop. Highest-leverage, lowest-risk FFI micro-opt.
  • Use DataView typed accessors, not hand-rolled byte math. On JSC DataView.getUint32(o,true) ties Buffer.readUInt32LE (~1.2 ns) and beats manual (b[o]|b[o+1]<<8|…) — the opposite of V8 lore. Big(U)int64Array views are the slowest u64 path (BigInt-box per element) and often illegal on unaligned Win32 structs — do not reach for them. JSC shows no measurable monomorphism penalty for Buffer/Uint8Array/subarray/DataView receivers — don't contort the design for it; spend the effort on alloc + bigint discipline.
  • Minimize FFI crossings and JSCallback invocations. The FFI floor is ~5 ns/call; a u64→bigint return adds ~10 ns. The whole product thesis is the macro-opt: ONE NtQuerySystemInformation for the whole process list, not N per-PID calls. A native→JS JSCallback costs ~39 ns/invocation — the ETW firehose must do the absolute minimum per callback (defer decode/format to a drain pass), never N callbacks where one buffered delivery would do. Hoist constant bigint args out of loops (a fresh BigInt() per call is ~16 ns). Preload then destructure hot exports (repo doctrine — captures the native fn, not the lazy wrapper; it is a correctness ordering, measured ~0 ns faster than the post-Preload static call, so don't expect speed from it — the levers above are where the time is).
  • Pace correctly. Async Bun.sleep(1) IS quantized to ~15.6 ms (≈30–60 Hz cap) — but Bun.sleepSync(1) is not (~1.5 ms ≈ 600 Hz), and CreateWaitableTimerExW gave no measurable edge over sleepSync at 1 ms. True 1000 Hz/1 ms with sub-10 µs jitter needs a busy-spin on Bun.nanoseconds() (pegs a core). sampler.ts must offer both: jitter-tolerant (sleepSync, low CPU) and precise (busy-spin, the literal 1000 Hz claim) — and the 1000 Hz number must be re-proven, not assumed from the timer.
  • Every hot path gets an isolated microbenchmark (Bun.nanoseconds(), 200k+ warm-up iterations to trigger JSC's JIT tiers, dead-code-elimination sink, median ns/op — the repo's terminal.bench.ts convention). Benchmarks DRIVE optimization at the unit level (per-row decode, per-tick sample, ticker jitter, single FFI call), not just produce a README table. A perf baseline is recorded in Phase 0 and gated at release (S14.x) exactly like a tsc error.
  • Prove it, don't hope — Bun ships bun:jsc. Because this is Bun-only, every hot path can be verified optimized, not assumed: numberOfDFGCompiles(fn) > 0 (timing optimized code, not the interpreter), profile(cb) tier breakdown (a warmed loop measured 99.3% FTL on this machine — assert FTL/DFG dominates), reoptimizationRetryCount(fn) (climbing = type-unstable deopt — fix the polymorphism, not the body), jscDescribe(obj) (same StructureID across the run = monomorphic shape), and heapStats().objectTypeCounts deltas (gate per-tick allocation at literally zero). The full verified toolkit + the measured shape/array/alloc/string levers are Appendix G.4/G.5 — use them in every microbench and audit step.
  • Don't satisfice — beating the baseline is the job, not meeting it. Correctness plus the known levers is the START of a hot-path audit, never the end. Once a path is correct, profile() it to find the ACTUAL dominant cost (often something this plan never listed), form a hypothesis, microbench an alternative, and keep it only if it wins AND stays correct, cast-free, and readable — then go again until returns genuinely diminish. Interrogate every read, branch, allocation, and pass: can it be hoisted, precomputed, cached, batched, fused into one pass, made lazy, or eliminated outright? Can the algorithm change, not just the constant factor? The S14.6 gate only enforces "no regression"; the standing goal is to beat the prior baseline every phase. Any lever you discover that this plan didn't list — prove it with the G.4 tools and add it to Appendix G.5, so the house standard compounds. Never stop at "fast enough."

Appendix F — Deviations from the gpu-plan template (for the next plan author)

This plan follows packages/gpu/PLAN.html's Appendix F contract. Where it deliberately diverges — and why — so the house standard evolves honestly:

  1. Zero binding/extraction risk, all decode risk. GPU extracted one 1,340-line engine; sysmon assembles ~12 modules from ~10 independent examples over an already-bound surface (dossier: 126/126 APIs bound). So the dominant risk shifts from "does the COM call work" to "is the struct offset right" — hence the PROVEN/UNVERIFIED split in Appendix B and the S3.4 offsets-referee + Phase 11 cross-validation that have no GPU analog.
  2. No all shim/migration phase. GPU's Phase 11 re-ran 30 demos through re-export shims because they shared helper modules. sysmon's examples are self-contained, so the regression analog is cross-validation against OS ground truth (tasklist/netstat/statfs/registry), not a demo sweep. No all/bun-win32 bump.
  3. Correctness is verified numerically against a SECOND source, not visually. GPU's doctrine is "look at the PNG." sysmon's is "two independent kernel sources must agree" (NtQSI vs PDH vs GetSystemTimes; snapshot vs GetProcessMemoryInfo; sysmon vs netstat) — a number that merely returns is not a number that is correct.
  4. Elevation + blocking are first-class API shapes, not afterthoughts. The ETW firehose is honestly a blocking, elevated, foreground call; the plan ships the no-admin census/tail as the lead surface and gates the firehose behind isElevated() with an actionable error — rather than pretending it's a clean AsyncIterable.
  5. Honest-scope against Node core is mandatory. Unlike GPU (where Node has no equivalent), Node's os/process/fs.statfs already cover the trivial metrics — the README must say what sysmon adds over core, or it overclaims. This is the one place the "everything they do, only better" framing needs the "core already does the easy part" caveat.
  6. Concurrency with a sibling plan. This plan was authored while the gpu plan was being executed in the same repo. Every phase pulls/rebases and never touches packages/gpu — a discipline the GPU plan (authored first) didn't need.
  7. Everything else is identical: ground-truth dossier → research-while-building doctrine → execute-time competitive intel with a parity matrix gated at release → bite-sized verified phases with a runtime proof per phase → pure-logic bun:test where FFI-free → AI.md docs with a run gate → one-OTP ordered publish → registry-poll + clean-room acceptance → final report with the deviation ledger.

G.3 — The discipline

  1. Baseline (S0.6b) records the floors on the release host before any work.
  2. Every hot-path phase (2, 3, 4, 6, 8) ends with a microbenchmark step that records ns/op and asserts zero per-tick/per-row/per-event allocation.
  3. Release gate (S14.6) re-runs everything and blocks release on a >~10% regression vs baseline AND on any forbidden hot-loop pattern (per-iteration Buffer.alloc/new DataView/typed-array, rebuilt BigInt(), re-read .ptr).
  4. README numbers come only from the gated run — never copied from this appendix.
  5. Generative pass (the real spirit — do this every hot-path phase). An audit does not just apply G.5's levers and check the gate. It profile()s the path to locate the true hot spot, then actively tries at least one optimization this plan did NOT anticipate — a fused pass, a precomputed table, a branch eliminated, a read avoided, an allocation hoisted, a better algorithm — measures it, and folds any proven win back into G.5. "Meets baseline" is the floor; the work is to beat it. Stop only at real diminishing returns, never at "good enough." The agent that ships the fastest correct path, not the one that merely passes the gate, did the job.
  6. Methodology for every microbench (all verified on Bun 1.4.0/JSC, this machine): Bun.nanoseconds(); warm ≥200k iterations (1k is NOT enough — numberOfDFGCompiles was still 0) to trigger JSC's tier-up; assert numberOfDFGCompiles(fn) > 0 and abort the bench if 0 (else you're timing LLInt/Baseline); fold results into a process-lived sink to defeat dead-code elimination; vary at least one offset per iteration to defeat loop-invariant hoisting; gcAndSweep()+fullGC() between (never within) trials; median of ≥5 trials. The repo's terminal.bench.ts is the convention to mirror.
